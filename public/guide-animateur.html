Gestion des fichiers

Les données du système sont réparties dans data/saisons/2025-2026/. Le comportement de chaque fichier diffère selon qu'on est en local ou sur Railway (production).

Fichier Railway au déploiement Rôle
équipes.json Toujours écrasé Noms, couleurs, priorité d'égalité
joueurs.json Toujours écrasé Liste des joueurs par équipe
séries.json Copié si absent Structure des séries — éditable via admin.html
parties.json Copié si absent Calendrier, liens Teams, matchups éliminatoires
thèmes.json Copié si absent Thèmes et sous-thèmes par questionnaire
questions.json Jamais touché Textes et réponses des questions
répondants.json Jamais touché Données accumulées en jeu — toute la saison
alignements.json Jamais touché Alignements des joueurs par partie
Règle clé : un déploiement Railway ne détruira jamais répondants.json ni alignements.json s'ils existent déjà sur le Volume. Les données de la saison sont protégées.

Volume des buzzers

Le volume des buzzers se règle dans marqueur.html via un curseur affiché dans la zone d'initialisation, avant de démarrer la partie.

  • 0 % — muet
  • 100 % — amplification par défaut (2× le volume système)
  • 200 % — amplification maximale

Deux boutons permettent de tester le son de chaque équipe une fois la partie sélectionnée, avant de la démarrer.

À tester en salle — le rendu varie beaucoup selon les haut-parleurs et l'acoustique.

📥 Importer un questionnaire → thèmes.json + questions.json

Source : fichier Excel avec une feuille nommée Vers BD  |  Effet : ajoute ou remplace le questionnaire correspondant

Le fichier doit contenir deux tableaux : tblThèmes (un thème par série) et tblQuestionnaires (une question par ligne).

Série 9 (Indice) : Q1 et Q2 sans réponse sont normales — la réponse n'est inscrite qu'à Q3.
Série 13 (Associations) : Q1 et Q2 contiennent les textes d'introduction des deux groupes.

📋 Éditeur de séries

La structure des séries est modifiable via l'éditeur intégré dans admin.html. Une fois la saison lancée (au moins une partie jouée), l'éditeur est verrouillé automatiquement.

  1. Modifier les séries via l'éditeur
  2. Sauvegarder (enregistré directement sur le serveur)
  3. Télécharger le JSON pour mettre à jour la copie locale

Répondants — deux fichiers, deux rôles

parties/répondants-{N}.json — fichier de travail en direct

Créé dès qu'un premier point est attribué durant la partie N. Mis à jour en temps réel :

  • Attribuer → ajoute une entrée
  • Annuler → supprime l'entrée
  • Écraser (nouvelle réponse sur une question déjà répondue) → remplace l'entrée après confirmation

Sert aussi à la reprise automatique : si le navigateur du marqueur plante ou est fermé, la reconnexion recharge l'état complet depuis ce fichier.

répondants.json — fichier cumulatif de la saison

Mis à jour uniquement au moment de terminer une partie. Alimente toutes les pages de statistiques (classement, compteurs, feuille de match, profil joueur).

Pour développeurs

Flux de déploiement : Local → GitHub → Railway

Chaque git push vers GitHub déclenche automatiquement un redéploiement sur Railway.

Réinitialisation du Volume (reset)

En cas de besoin, le script init-volume.js permet de réinitialiser le Volume depuis les fichiers git :

# Sans --force : copie seulement les fichiers absents (sécuritaire)
railway run node init-volume.js

# Avec --force : écrase TOUT depuis git (⚠️ détruit les données de jeu)
railway run node init-volume.js --force
Attention : --force écrase répondants.json et alignements.json. À n'utiliser qu'en début de saison ou pour un reset complet.

Workflow séries en début de saison

1. Modifier séries.json via admin.html
2. Sauvegarder (enregistré sur le serveur)
3. Télécharger le JSON → remplacer data/saisons/AAAA/séries.json en local
4. git push → mis à jour dans git pour la prochaine saison

🗓️ Démarrer une nouvelle saison

Étapes à suivre

  1. Créer la nouvelle saison dans la page Administration → section Gestion des saisons. Entrer le format AAAA-AAAA (ex: 2026-2027) et cliquer Créer.
  2. Activer la nouvelle saison : sélectionner la saison dans le menu déroulant et cliquer Activer. Toutes les parties suivantes utiliseront cette saison.
  3. Configurer les équipes et joueurs dans la section Équipes & Joueurs de la page Administration.
  4. Importer le calendrier (fichier parties.json) dans la section Calendrier.
  5. Configurer la structure des séries dans la section Séries de la page Administration (types de séries, nombre de questions, points, etc.).
  6. Importer les questionnaires au fur et à mesure des parties : avant chaque partie, importer le fichier questions.xlsx correspondant dans la section Questionnaire.

Notes importantes

  • Chaque saison est isolée dans son propre dossier (data/saisons/AAAA-AAAA/). Les données de saisons précédentes ne sont pas affectées.
  • Les statistiques des saisons passées restent consultables en sélectionnant la saison dans le menu déroulant des pages de statistiques.
  • Si la saison est déployée sur Railway, les fichiers sont stockés sur le Volume. La saison active est persistée dans data/saison-active.txt sur le Volume — elle survit aux redémarrages.
  • Pour revenir à une saison précédente (ex: corriger des données), changer la saison active dans Administration puis revenir à la saison courante quand terminé.

Sur Railway (production)

La variable d'environnement SAISON_ACTIVE est un fallback — elle n'est lue qu'au démarrage si data/saison-active.txt est absent. En utilisant l'interface Administration, saison-active.txt est mis à jour automatiquement et aucun redéploiement n'est nécessaire.
