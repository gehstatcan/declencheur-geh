Déroulement d'une partie

1. Initialisation

Accéder à marqueur.html. Le bandeau orange 🔧 Initialisation indique qu'aucune partie n'est en cours.

  1. Sélectionner la partie dans la liste déroulante. Seules les parties des 5 derniers jours apparaissent — cocher Afficher toutes les parties si nécessaire.
  2. Choisir le mode de placement des joueurs à l'écran :
    • 🎲 Positions aléatoires — les joueurs sont disposés aléatoirement dans leur bloc d'équipe.
    • 🔢 Positions fixes — les joueurs sont placés selon le champ Position défini pour chaque joueur dans l'écran "administration".
  3. Cliquer ▶️ Démarrer la partie.
Si tous les joueurs ne sont pas encore connectés : cliquer sur Démarrer quand même. Les joueurs peuvent rejoindre en cours de route — ils s'enregistrent dès qu'ils choisissent leur nom. Aucune conséquence sur le déroulement.

2. L'écran central — à surveiller en permanence

Le bloc central affiche en tout temps la série, le thème et le numéro de question (ex. Série 1 — Individuelle / Langue / Question 1 / 8 — 10 pts). C'est la référence absolue pour savoir où vous en êtes.

Il est essentiel que cet affichage corresponde à la question lue à voix haute. La feuille de match est générée à partir de ces données — une question accordée à la mauvaise série ou au mauvais numéro faussera les statistiques.

Avant d'attribuer un point, vérifiez toujours que la série et le numéro de question affichés correspondent bien à ce que vous venez de lire.

3. Flux d'une question — boutons principaux

Dès qu'un joueur appuie, son nom s'affiche dans la zone buzzer. Trois actions sont possibles :

BoutonEffet
✅ 10 pts Attribue 10 pts à l'équipe du joueur et passe automatiquement à la question suivante.
✅ 5 pts Visible uniquement pour certaines questions (voir séries spéciales ci-dessous). Attribue 5 pts et avance.
❌ Mauvaise Enregistre une mauvaise réponse. La question reste active — l'équipe adverse peut buzzer.
🔄 Reset Réinitialise le buzzer sans enregistrer de réponse. Utile si un joueur a buzzé par erreur avant que la question soit lue.
🗑️ Annuler la réponse Supprime le dernier point accordé (la question précédente). À utiliser si une erreur est constatée juste après avoir cliqué 10 pts.
Donner des points sans buzzer : si personne n'a buzzé mais qu'une équipe ou un joueur mérite des points (ex. : réponse collective donnée à voix haute), cliquer sur le bon joueur puis sur ✅ 10 pts — si c'est une question d'équipe, le système attribue les points à l'équipe.

4. Naviguer entre les questions et les séries

  • ◀️ Question précédente / Question suivante ▶️ — avance ou recule d'une question dans la série.
  • ⏮️ Série précédente / Série suivante ⏭️ — saute à la première question de la série adjacente.
  • 🔄 Retour initialisation — revient à l'écran de sélection sans perdre les données. La partie peut être relancée exactement où elle en était.

5. Séries spéciales — points différents

Pour la majorité des séries, le marqueur gère automatiquement les points. Deux séries font exception :

SérieQuestionsParticularité
Série 6 (Les 4 V) Q5 à Q8 Le bouton ✅ 5 pts apparaît. Les bonnes réponses à ces questions valent 5 pts (au lieu de 10).
Série 13 (Associations) Q4 et Q8 Valent 20 pts. Toutes les autres questions valent 10 pts.
Série 9 (Indice) : Q1 et Q2 n'ont pas de réponse associée dans le gabarit — c'est normal, la réponse n'est inscrite qu'à Q3. Si l'indice est réussi à Q1 ou Q2, utiliser le bouton Série suivante ⏭️ pour sauter les questions restantes et passer à la série suivante.

6. Timer de réponse (reset auto)

La case ⏱️ Reset auto après X sec est cochée par défaut (8 secondes). Une fois un buzzer reçu, si aucun bouton n'est cliqué dans le délai, le buzzer se réinitialise automatiquement — pratique pour les moments où personne ne répond.

  • Le délai est ajustable (1–30 secondes) dans le champ numérique adjacent.
  • Décocher la case pour désactiver entièrement le reset automatique.

7. Volume des buzzers et test des sons

Le curseur 🔊 Volume buzzers est accessible une fois la partie démarrée (pas avant). Par défaut à 200% (amplification maximale). Ajuster selon l'acoustique de la salle.

  • Les boutons 🔔 Équipe A et 🔔 Équipe B permettent de tester le son de chaque équipe à tout moment pendant la partie.
  • 0% = muet, 100% = amplification normale, 200% = maximum.

8. Affichages optionnels

  • 📊 Afficher pts/joueur — affiche les points individuels de chaque joueur dans les blocs d'équipe.
  • ⚡ Temps de réaction — affiche le temps en secondes entre le début de la question et le buzzer.

8. Crash ou déconnexion — pas de panique

Chaque réponse accordée est sauvegardée immédiatement sur le serveur dans un fichier temporaire (répondants-{N}.json). Si le navigateur de l'animateur plante ou si l'onglet est fermé accidentellement :

  1. Retourner sur marqueur.html et sélectionner la même partie.
  2. Cliquer ▶️ Démarrer la partie — toutes les réponses déjà enregistrées sont rechargées automatiquement.
  3. Naviguer jusqu'à la bonne question et reprendre.

Pour les joueurs, le même mécanisme s'applique : si leur onglet se ferme, ils retournent sur joueur.html, rejoignent la même partie et choisissent à nouveau leur nom. Ils retrouvent leur équipe en temps réel.

9. Téléchargements en cours de partie

Deux boutons de sauvegarde sont disponibles pendant le jeu :

  • 💾 Télécharger partie en cours — exporte les répondants de la partie active seulement (fichier répondants-{N}.json).
  • 💾 Télécharger cumulatif — exporte le fichier complet de la saison (répondants.json) tel qu'il est au moment du téléchargement.
Ces téléchargements sont optionnels — une sauvegarde ZIP complète est proposée automatiquement à la fin de chaque partie.

10. Terminer la partie

Cliquer 🏁 Terminer la partie lorsque toutes les séries sont complétées. Le système consolide les données dans répondants.json et affiche une bannière de sauvegarde. Une sauvegarde ZIP est recommandée à ce moment.

Gestion des fichiers

Les données du système sont réparties dans data/saisons/2025-2026/. Le comportement de chaque fichier diffère selon qu'on est en local ou sur Railway (production).

Fichier Railway au déploiement Rôle
équipes.json Copié si absent Noms, couleurs, priorité d'égalité — éditable via admin.html
joueurs.json Copié si absent Liste des joueurs par équipe — éditable via admin.html
séries.json Copié si absent Structure des séries — éditable via admin.html
parties.json Copié si absent Calendrier, liens Teams, matchups éliminatoires
thèmes.json Jamais touché Thèmes et sous-thèmes — géré via import questionnaire (admin.html)
questions.json Jamais touché Textes et réponses des questions — géré via import questionnaire (admin.html)
répondants.json Jamais touché Données accumulées en jeu — toute la saison
alignements.json Jamais touché Alignements des joueurs par partie
Règle clé : un déploiement Railway ne détruira jamais répondants.json ni alignements.json s'ils existent déjà sur le Volume. Les données de la saison sont protégées.

Volume des buzzers

Le volume des buzzers se règle dans marqueur.html via un curseur affiché dans la zone d'initialisation, avant de démarrer la partie.

  • 0 % — muet
  • 100 % — amplification par défaut (2× le volume système)
  • 200 % — amplification maximale

Deux boutons permettent de tester le son de chaque équipe une fois la partie sélectionnée, avant de la démarrer.

À tester en salle — le rendu varie beaucoup selon les haut-parleurs et l'acoustique.

📋 Éditeur de séries

La structure des séries est modifiable via l'éditeur intégré dans admin.html. Une fois la saison lancée (au moins une partie jouée), l'éditeur est verrouillé automatiquement.

  1. Modifier les séries via l'éditeur
  2. Sauvegarder (enregistré directement sur le serveur)
  3. Télécharger le JSON pour mettre à jour la copie locale

Répondants — deux fichiers, deux rôles

parties/répondants-{N}.json — fichier de travail en direct

Créé dès qu'un premier point est attribué durant la partie N. Mis à jour en temps réel :

  • Attribuer → ajoute une entrée
  • Annuler → supprime l'entrée
  • Écraser (nouvelle réponse sur une question déjà répondue) → remplace l'entrée après confirmation

Sert aussi à la reprise automatique : si le navigateur du marqueur plante ou est fermé, la reconnexion recharge l'état complet depuis ce fichier.

répondants.json — fichier cumulatif de la saison

Mis à jour uniquement au moment de terminer une partie. Alimente toutes les pages de statistiques (classement, compteurs, feuille de match, profil joueur).

Sauvegarde ZIP

Le système offre deux façons de télécharger une sauvegarde complète de tous les fichiers JSON de la saison active (équipes, joueurs, parties, séries, questions, thèmes, répondants, alignements).

Depuis le marqueur (après une partie)

Une bannière apparaît automatiquement à la fin de chaque partie avec un bouton Télécharger la sauvegarde ZIP. L'animateur peut ainsi conserver une copie locale sans avoir accès à l'administration.

Depuis l'administration

La section Gestion des saisons dans admin.html offre le même bouton de téléchargement, accessible à tout moment.

Usage développement : télécharger le ZIP de Railway et extraire les fichiers dans data/saisons/2025-2026/ en local permet de reproduire exactement l'environnement de production.

Réinitialisation du Volume (reset)

En cas de besoin, le script init-volume.js permet de réinitialiser le Volume depuis les fichiers git :

# Sans --force : copie seulement les fichiers absents (sécuritaire)
railway run node init-volume.js

# Avec --force : écrase TOUT depuis git (⚠️ détruit les données de jeu)
railway run node init-volume.js --force
Attention : --force écrase répondants.json et alignements.json. À n'utiliser qu'en début de saison ou pour un reset complet.

Workflow séries en début de saison

1. Modifier séries.json via admin.html
2. Sauvegarder (enregistré sur le serveur)
3. Télécharger le JSON → remplacer data/saisons/AAAA/séries.json en local
4. git push → mis à jour dans git pour la prochaine saison

🗓️ Démarrer une nouvelle saison

Étapes à suivre

  1. Créer la nouvelle saison dans la page Administration → section Gestion des saisons. Entrer le format AAAA-AAAA (ex: 2026-2027) et cliquer Créer.
  2. Activer la nouvelle saison : sélectionner la saison dans le menu déroulant et cliquer Activer. Toutes les parties suivantes utiliseront cette saison.
  3. Configurer les équipes et joueurs dans la section Équipes & Joueurs de la page Administration.
  4. Importer le calendrier (fichier parties.json) dans la section Calendrier.
  5. Configurer la structure des séries dans la section Séries de la page Administration (types de séries, nombre de questions, points, etc.).
  6. Importer les questionnaires au fur et à mesure des parties : avant chaque partie, importer le fichier questions.xlsx correspondant dans la section Questionnaire.

Notes importantes

  • Chaque saison est isolée dans son propre dossier (data/saisons/AAAA-AAAA/). Les données de saisons précédentes ne sont pas affectées.
  • Les statistiques des saisons passées restent consultables en sélectionnant la saison dans le menu déroulant des pages de statistiques.
  • Si la saison est déployée sur Railway, les fichiers sont stockés sur le Volume. La saison active est persistée dans data/saison-active.txt sur le Volume — elle survit aux redémarrages.
  • Pour revenir à une saison précédente (ex: corriger des données), changer la saison active dans Administration puis revenir à la saison courante quand terminé.

Sur Railway (production)

La variable d'environnement SAISON_ACTIVE est un fallback — elle n'est lue qu'au démarrage si data/saison-active.txt est absent. En utilisant l'interface Administration, saison-active.txt est mis à jour automatiquement et aucun redéploiement n'est nécessaire.

📤 Importer des données historiques sur Railway

En temps normal, répondants.json et alignements.json sont ignorés par git (skip-worktree) pour éviter d'écraser accidentellement les données de jeu sur le Volume. Mais il est possible de les pousser ponctuellement via git.

Pourquoi skip-worktree et pas .gitignore ?

.gitignore empêche git de jamais suivre un fichier — impossible de le pousser ensuite. skip-worktree dit à git d'ignorer les modifications locales au quotidien, tout en gardant le fichier suivi. On peut lever le flag temporairement pour faire un push, puis le remettre.

Procédure d'import

  1. Générer les fichiers localement (ex: script generer-saison24.js)
  2. Lever le flag skip-worktree sur les deux fichiers : 
    git update-index --no-skip-worktree "data/saisons/AAAA-AAAA/répondants.json"
git update-index --no-skip-worktree "data/saisons/AAAA-AAAA/alignements.json"
  3. Dans initialiserVolume() de index.js, ajouter temporairement une section d'écrasement forcé pour ces deux fichiers : 
    // TEMPORAIRE — retirer après déploiement
["répondants.json", "alignements.json"].forEach((f) => {
  const src = path.join(__dirname, "data", "saisons", saisonActive, f);
  const dst = path.join(dossierSaison, f);
  if (fs.existsSync(src)) fs.copyFileSync(src, dst);
});
  4. Commit + push → Railway redéploie et copie les fichiers sur le Volume
  5. Dès que Railway confirme le déploiement, revert immédiatement : 
    git revert HEAD --no-edit
git update-index --skip-worktree "data/saisons/AAAA-AAAA/répondants.json"
git update-index --skip-worktree "data/saisons/AAAA-AAAA/alignements.json"
git push
Important : ne pas laisser le bloc d'écrasement forcé plus longtemps que nécessaire — chaque redéploiement écraserait le Volume avec la version git, effaçant toutes les nouvelles parties jouées.
